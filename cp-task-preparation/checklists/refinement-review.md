# Refinement Review Checklist

- The refinement mode is identified: in-session or post-session.
- The refinement report identifies the mode and cause category.
- In-session refinements fixed or stabilized the current package/task behavior first.
- In-session changes are minimal and targeted unless the issue is clearly architectural.
- When both occurred, package correction is separated from reusable skill correction.
- End-of-session refinement runs Track A and Track B separately.
- Track A findings include lesson, evidence, canonical owner, cause category, and status.
- Track B findings include lesson, evidence, canonical owner, and status, or explicitly state no new reusable lessons.
- Workflow/package-process issues are not mixed into Track B unless they directly affect testcase quality.
- Each finding has concrete session evidence.
- Each finding is classified with `references/issue-taxonomy.md`.
- One-off mistakes are separated from reusable methodology.
- Existing canonical owners were searched before adding new guidance.
- The proposed change has one owner file.
- Checklist additions are action gates, not methodology prose.
- Failure-mode additions are reusable mechanisms, not anecdotes.
- The update includes a retrieval and regression check.
